# AI 测试文案：假勤、组织、绩效分阶段验收

## 1. 文档目标

本文档用于指导 AI 在 `People Ops` 项目中按优先级分阶段完成测试和验收。

测试顺序必须与开发顺序保持一致：

1. 假勤
2. 组织
3. 绩效

这样做的目的是让 AI 优先聚焦当前成熟度最高的模块，避免在绩效尚未真正成型前，把未完成范围和实际缺陷混在一起。

## 2. 核心测试原则

AI 测试时必须遵守以下规则：

1. 测的是业务行为，不只是接口返回成功。
2. 必须验证“本地优先”：查询页面和计算接口不能依赖实时钉钉响应。
3. 必须验证同步、发放、追溯、匹配、重试等流程的幂等性。
4. 必须验证关键创建、修改、提交、确认、重试、人工修正动作的审计能力。
5. 必须验证权限边界和数据范围边界。
6. 结果输出以高价值问题为主，不要只给泛泛而谈的测试总结。
7. 当某阶段只做了一部分时，必须明确区分：
   - 已通过的行为
   - 缺失的行为
   - 高风险假设

## 3. 现有测试基础

现有项目已具备以下测试基础：

- 后端测试：`internal/**/*_test.go`
- API 契约测试：`tests/api-contract-test.go`
- 钉钉 mock：`tests/mock/dingtalk_mock.go`
- 现有前端 E2E 示例：
  - `frontend/tests/e2e/login.spec.ts`
  - `frontend/tests/e2e/attendance.spec.ts`
  - `frontend/tests/e2e/organization.spec.ts`

AI 测试时应优先沿用这些结构和模式，而不是另起一套完全无关的测试体系。

## 4. 测试顺序与范围

### 阶段一：假勤

这个阶段应最先测试，而且要测得最深，因为它离可用生产范围最近。

#### 4.1 必测范围

1. 批量排班预览与应用
2. 大小周日历正确性
3. 节假日豁免正确性
4. 排班下发钉钉与失败重试
5. 考勤统计正确性
6. 加班审批匹配正确性
7. 调休额度生成正确性
8. 调休重放到钉钉行为
9. 年假资格重算
10. 年假季度发放
11. 年假追溯补发
12. 年假消费记录和余额行为

#### 4.2 必测场景

1. 正常路径
2. 空数据场景
3. 非法输入
4. 重复执行
5. 部分失败后的重试
6. 操作员可见的错误提示
7. 权限限制
8. 需要留痕的动作是否产生审计记录

#### 4.3 关键断言

1. 排班预览结果必须与规则期望一致，再允许应用。
2. 大小周加节假日规则不能产生错误的必打卡要求。
3. 已审批但没有有效打卡的加班，不能生成调休额度。
4. 重跑匹配不能重复增加余额。
5. 重跑季度发放不能产生重复台账。
6. 只有资格变化满足条件时，才允许生成追溯补发。
7. 即使钉钉不可用，本地查询接口仍应基于本地数据工作。

### 阶段二：组织

这个阶段主要验证管理视角是否建立在正确的本地组织数据之上。

#### 4.4 必测范围

1. 组织树加载
2. 员工列表筛选
3. 员工档案查询和编辑
4. 入职、调岗、离职列表行为
5. 团队概览卡片
6. 组织驾驶舱筛选
7. 结构分析图表
8. 生日和入职周年提醒
9. 数据范围限制

#### 4.5 必测场景

1. 部门筛选是否正确影响结果
2. 本地档案字段缺失时是否优雅降级
3. 无组织数据场景
4. 钉钉同步数据和本地补录数据混合场景
5. 主管视角与 HR 视角差异
6. 敏感字段访问限制

#### 4.6 关键断言

1. 组织驾驶舱必须基于本地同步数据和本地档案数据生成。
2. 结构分析使用的字段和口径必须正确。
3. 员工档案更新后应正确落库，且不能破坏钉钉标识字段。
4. 团队概览指标必须与底层员工和流动数据一致。
5. 用户只能看到自己权限范围内的数据。

### 阶段三：绩效

这个阶段必须按“新子系统”来测试，不能当成现有人才分析页面的小增强。

#### 4.7 必测范围

1. 绩效活动创建与生命周期
2. 周期配置
3. 模板配置
4. 评审关系正确性
5. 自评提交流程
6. 主管评分与评语
7. 进度看板
8. 归档行为
9. 批量评审与批量确认
10. 强制分布规则配置
11. 强制分布提交拦截
12. 配额进度看板
13. C/D 面谈强制要求
14. 钉钉提醒触发
15. 审计日志

#### 4.8 必测场景

1. 正常自评到主管评审的流转
2. 缺失必填评审数据
3. 重复提交
4. 超配额等级分布
5. 公司、部门、团队不同范围规则
6. 批量评审中的部分失败
7. C/D 员工无面谈记录
8. 提醒发送失败但业务数据不丢失
9. 权限越权

#### 4.9 关键断言

1. 自评和主管评审的状态流转必须合法且可追踪。
2. 强制分布规则必须拦截非法提交，并返回清晰提示。
3. 不同范围的规则必须作用在正确的人群上。
4. 批量操作必须安全、可追踪，并支持部分恢复。
5. C/D 员工不能绕过面谈要求。
6. 进度看板必须准确反映底层评审状态。
7. 所有关键动作都必须留下系统日志。

## 5. 推荐测试分层

AI 测试时应把工作拆到以下层次：

1. 后端单测：规则引擎、计算逻辑
2. 后端集成测试：repository、service 编排
3. API 契约测试：请求、响应、校验行为
4. 前端页面/组件行为测试
5. Playwright E2E：核心操作流程
6. 人工验收风格检查单：高风险业务流程

## 6. 每阶段必须输出的验收清单

对于每个阶段，AI 测试报告必须明确回答：

1. 哪些行为已完整通过
2. 哪些行为只部分通过
3. 哪些行为尚未实现
4. 是否产生审计日志
5. 是否满足权限要求
6. 是否保持本地优先
7. 重跑和重试是否幂等
8. 还存在哪些生产风险

## 7. 可直接投喂 AI 的测试提示词

以下内容可以直接复制给其他 AI 测试助手。

```text
你正在测试 People Ops 项目。

请严格按以下顺序测试：
1. 假勤
2. 组织
3. 绩效

通用规则：
- 重点验证真实业务行为，不只是接口是否成功。
- 必须验证本地优先。查询页面和计算 API 不能依赖实时钉钉响应。
- 重点关注幂等性、审计日志、权限边界、操作员可见的错误处理。
- 如果一个功能尚未完成，要把“未实现”与“真实缺陷”分开。

阶段一：假勤
- 测试批量排班、大/小周规则、节假日豁免、排班下发流程、加班匹配、调休生成、年假资格重算、季度发放、追溯补发、消费记录。

阶段二：组织
- 测试组织树、员工列表筛选、员工档案、入转调离流程、团队概览驾驶舱、结构分析、提醒卡片、数据范围限制。

阶段三：绩效
- 把绩效视为新子系统。
- 测试绩效活动、周期、模板、自评、主管评审、进度跟踪、归档、强制分布规则、提交拦截、批量评审、C/D 面谈要求、提醒行为和审计日志。

每次输出都必须包含：
- 已通过行为
- 失败行为
- 缺失行为
- 剩余风险
```

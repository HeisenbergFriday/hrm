# AI 开发文案：假勤、组织、绩效分阶段实施

## 1. 文档目标

本文档用于指导 AI 在 `People Ops` 项目中按优先级分阶段完成开发，并确保实现路径与当前项目现状匹配。

本次交付顺序必须严格遵循以下优先级：

1. 假勤
2. 组织
3. 绩效

原因如下：

- 假勤模块现有基础最好，最接近可直接完善上线。
- 组织模块已经具备钉钉同步、员工档案、入转调离等基础，但管理驾驶舱和结构分析还不完整。
- 绩效模块目前只有较薄的结果字段和展示页面，应视为新子系统建设，而不是小改动。

本文档的重点不是抽象业务描述，而是给 AI 明确开发边界、代码入口、拆分方式、验收要求，以及可直接投喂的实现提示词。

## 2. 核心约束

AI 开发时必须遵守以下规则：

1. 优先复用现有“本地优先”架构，不要引入新的实时钉钉依赖。
2. 不要在查询页面、看板接口、计算接口、列表接口、详情接口中新增实时钉钉调用。
3. 钉钉主要用于登录、组织同步、审批流、待办和通知，不应成为运行时业务计算依赖。
4. 业务计算必须以本地表为准。
5. 优先复用现有 `router`、`service`、`repository`、`model` 结构，不要平行再造一套系统。
6. 除非需求明确要求调整，否则不要破坏现有年假、调休、排班、大/小周逻辑。
7. 所有关键写操作都必须可审计、可追溯。
8. 新增 API 应保持现有 `/api/v1/...` 风格，并兼容当前 `frontend/src/services/api.ts` 的使用方式。
9. 所有批处理、重算、同步补偿、重试动作都必须幂等。
10. 对于尚未确认的业务口径，优先做成可配置策略，并在实现说明中明确假设。

## 4. 交付范围与顺序

### 阶段一：先做假勤

这个阶段必须最先完成，因为当前完成度最高、复用价值最大，而且已经有较多可直接完善上线的代码基础。

#### 4.1 本阶段范围

1. 批量排班能力产品化
   - 支持按规则批量为多人配置打卡周期、打卡日期、班次与排班方案。
   - 支持预览、校验、批量下发、失败重试、同步日志。
2. 大小周规则自动化
   - 支持企业大小周作息规则配置。
   - 自动识别法定节假日与大周周末休息时段。
   - 自动豁免无需打卡的时段，避免无效提醒与误判。
3. 加班全流程线上闭环
   - 员工发起加班审批。
   - 同步审批结果到本地。
   - 用本地打卡记录核验有效加班时长。
   - 自动生成调休额度、台账、失败重试记录、人工修正记录、审计日志。
4. 年假季度分期发放
   - 支持按自然季度发放年假。
   - 支持按工龄配置年度总天数与季度发放比例。
   - 支持试用期在职时段纳入应发资格。
   - 支持员工转正后追溯补算历史季度应发额度。
   - 支持资格重算、发放台账、消费记录和余额核对。
5. 异常识别与运维可视化
   - 考勤异常识别与展示补强。
   - 同步任务、发放任务、匹配任务的日志、错误反馈和管理员可见性补强。

#### 4.2 落地分工

钉钉负责：

- 班次
- 考勤组
- 打卡原始记录
- 加班审批流
- 消息提醒
- 假期余额承载与余额回写

本地系统负责：

- 批量排班规则、排班预览、批量下发
- 大小周规则引擎、节假日豁免、异常判断
- 加班审批同步、本地打卡核验、有效时长计算、调休额度生成
- 年假季度发放规则、工龄规则、资格重算、试用期纳入、转正追溯补算
- 发放台账、消费记录、同步日志、失败重试、人工修正、审计日志

#### 4.3 开发指导

AI 应优先补强和收口现有能力，而不是推翻重做：

- 以现有大小周、节假日逻辑为主线继续完善。
- 以现有年假资格、季度发放、追溯补发逻辑为主线继续完善。
- 以现有加班匹配和钉钉余额回写逻辑为主线继续完善。

应重点补这些方面：

1. 规则配置体验和 API 清晰度
2. 批量操作的健壮性
3. 校验与操作反馈
4. 异常处理和重试流程
5. 发放、追溯、加班同步、排班下发等动作的审计可见性

#### 4.4 本阶段验收标准

1. 批量排班可对多个员工进行预览和应用，并能提供明确的失败原因。
2. 大小周加法定节假日规则能生成正确的本地日历结果。
3. 加班匹配仍然保持本地优先和幂等。
4. 调休额度生成与回滚可追溯，不能因重跑造成重复余额。
5. 年假季度发放、资格重算、转正追溯补算结果可追溯。
6. 新增页面查询不依赖实时钉钉调用。

### 阶段二：再做组织

这个阶段是在现有组织与员工基础上补齐管理视角能力。

#### 4.5 本阶段范围

1. 组织基础管理与团队概览
   - 同步部门、员工、汇报关系、岗位信息。
   - 首页展示团队在职人数、直属下级、试用期、转正预警、绩效结果占位、流动情况等。
   - 支持按公司 / 部门 / 团队查看团队规模、结构、变动趋势。
2. 管理视角看板与多维筛选
   - 用统计卡、图表、排行榜展示关键指标。
   - 支持按组织、时间、员工类型、岗位、职级等维度筛选。
3. 人才结构分析
   - 正式 / 试用 / 实习 / 劳务 / 兼职等类型分布。
   - 年龄、司龄、学历、职级结构分析。
   - 管理人员、专业人员、技术人员等结构占比分析。
4. 入转调离管理
   - 新员工入职台账、试用期跟踪、转正节点提醒。
   - 计划转正、实际转正记录。
   - 调岗、调部门、汇报关系变化留痕。
5. 数据范围与权限补强
   - 按 HR、主管、普通管理者视角控制数据范围。
   - 看板、列表、档案、趋势图都要遵守同一套权限边界。

#### 4.6 数据指导与范围边界

以下数据应以钉钉同步结果为主：

- 部门
- 员工
- 汇报关系
- 岗位信息
- 部分入职和转正相关日期

以下数据应依赖本地档案表补充：

- 学历
- 专业
- 毕业院校
- 职级补充字段
- 钉钉不稳定或不完整的人事扩展字段

以下内容默认不作为本阶段首版强制范围，除非项目里已有稳定主数据来源：

- 编制
- 预算
- 项目成本
- 薪酬成本

不要假设只靠钉钉就能形成完整的组织分析数据集。

#### 4.7 本阶段验收标准

1. 管理者可按公司、部门或团队查看组织概览。
2. 结构分析图表来源于本地同步数据和本地员工档案数据。
3. 员工档案支持本地补录，补齐钉钉不足字段。
4. 驾驶舱支持按组织、时间等维度筛选。
5. 权限边界和数据范围边界持续有效。
6. 入转调离、提醒、概览指标与底层数据口径保持一致。

### 阶段三：最后做绩效

这个阶段必须按“新子系统”方式建设。

#### 4.8 本阶段范围

1. 绩效全流程线上闭环
   - 员工在线填写自评内容。
   - 主管在线完成评审打分与评语填写。
   - 修改、提交、审批、确认等操作全部留存系统日志。
   - 绩效结果可归档、可追溯、可查询。
2. 绩效基础主数据
   - 绩效活动
   - 绩效周期
   - 绩效模板
   - 评分关系
   - 进度看板
3. 绩效等级强制分布自动化
   - 支持 S/A/B/C/D 等级配额规则配置。
   - 支持公司 / 部门 / 团队层级差异化配置。
   - 部门提交绩效结果时，系统实时核算等级分布占比。
   - 超出预设配额时拦截提交并返回清晰提示。
   - 仅在调整到符合规则后允许最终提交。
4. 流程效率优化
   - 支持主管批量评审、批量确认。
   - 支持对待办和临近截止节点做钉钉提醒。
   - 仅对 C/D 级员工强制要求发起绩效面谈流程。
5. 低绩效员工跟踪
   - 记录 C/D 面谈发起、过程、结果、后续跟踪动作。
   - 面谈要求应成为提交校验的一部分，而不是线下口头要求。
6. 电子确认与存证边界
   - 如果只是“系统确认留痕”，首版由系统日志和确认记录承担。
   - 如需电子签章或法律级存证，视为第三方集成，不作为当前首版默认前提。

#### 4.9 首版绩效暂不纳入范围

除非用户后续明确升级范围，首版不包含：

1. 完整 OKR 或组织-个人目标对齐系统
2. 目标全周期跟踪与拆解
3. 复杂薪酬联动
4. 预算联动
5. 项目成本联动
6. 外部电子签章深度集成

说明：

- “组织 - 个人目标对齐与全周期追踪”目前属于待调研事项，不应卡住当前绩效闭环开发。
- “配额进度可视化看板”可以先预留数据结构和接口，不必阻塞首版主流程上线。

#### 4.10 落地分工与架构指导

钉钉负责：

- 表单
- 审批流
- 待办
- 通知
- 员工自评提交入口
- 主管确认提交入口

本地系统负责：

- 绩效活动、周期、模板、评分模型主数据
- 评审关系、评审进度、结果归档
- 自评内容与主管评审内容落库
- 批量评审与批量确认
- 强制分布规则引擎
- 公司 / 部门 / 团队级配额规则配置
- 提交校验与提交拦截
- C/D 面谈记录与跟踪
- 报表与看板
- 操作审计

第三方可选：

- 电子签章
- 存证

如果只是“系统确认留痕”，首版由本地系统完成，不强制引入第三方。

不要把绩效流程硬塞进现有 `TalentAnalysis` 表。
必须为流程型绩效系统建立独立模型、状态流转和服务层。

#### 4.11 关键规则与实现要求

1. 绩效应被建模为“流程系统”而不是单张结果表。
2. 自评、主管评审、提交确认、归档必须有明确状态机。
3. 系统必须保存每次修改、提交、退回、审批、确认的操作记录。
4. 强制分布规则必须是可配置规则，不要把比例写死在代码里。
5. 当前业务示例中的 `S/A/B/C/D = 10%/20%/40%/10%/5%` 合计为 85%，实现时必须按“可配置配额”处理，不要假设配额之和一定等于 100%。
6. 如果后续业务要求“配额总和必须等于 100%”，应在配置层做显式校验，并给出清晰错误提示。
7. 低绩效面谈必须作为 C/D 结果提交前的校验条件之一。
8. 钉钉提醒发送失败时，不应影响本地绩效数据落库。
9. 所有查询、看板、归档接口必须基于本地数据，不依赖实时钉钉。

#### 4.12 本阶段验收标准

1. 绩效周期、模板、评审关系可在本地配置和维护。
2. 员工自评与主管评审可以完成闭环，并留下完整日志。
3. 批量评审与批量确认具备幂等和部分失败可恢复能力。
4. 强制分布规则支持按范围配置，并在提交时生效。
5. 超配额提交会被拦截，并返回清晰提示。
6. C/D 面谈要求可被强制执行并可追溯。
7. 评审进度、归档结果、配额使用情况都可从本地查询。

## 5. 推荐实施策略

AI 必须严格按以下顺序工作：

1. 先把假勤产品化收口和补强完成。
2. 再在现有同步数据基础上建设组织驾驶舱和结构分析。
3. 最后单独建设绩效子系统。

每个阶段内部的建议顺序：

1. 先从本地数据模型和业务服务开始
2. 再补 repository
3. 再补 API
4. 再补前端页面或页面增强
5. 最后补测试并完成自检

如果某个阶段存在明显待确认点，不要因为业务细节不明就停住整体建设，应先做：

1. 本地模型
2. 状态流转
3. 规则配置能力
4. 可灰度启用的界面与接口

## 6. 必须交付的内容

每个阶段至少要交付：

1. 数据模型变更
2. 后端 service 与 repository
3. API handlers 与 routes
4. 前端页面、组件、API client
5. 测试代码
6. 简短实现说明，说明假设、边界和剩余风险

对于复杂子系统，AI 还应补充：

1. 状态机说明
2. 幂等策略说明
3. 审计策略说明
4. 批量操作失败恢复策略说明

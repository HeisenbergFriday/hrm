# AI 开发工作流

## 重要提醒

**本文档不是日常任务的默认必读文档。**

日常开发时，`CLAUDE.md` 是唯一默认读取文档。

只有在以下情况才读取本文档：
- 初始化 / 重构 AI 文档体系
- 用户要求更新 AI 项目上下文文档
- 用户要求根据 git diff 更新 AI 文档
- 用户要求查看、修改或解释 AI 工作流
- 修改了 AI 协作规则、读取策略或文档分层规则
- 当前流程不明确，仅靠 `CLAUDE.md` 无法判断下一步

---

## 开发生命周期概览

完整的开发生命周期包括 6 个阶段、16 个步骤：

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：需求分析                                             │
│  1. 识别需求意图                                              │
│  2. 声明读取计划                                              │
│  3. 按需读取上下文                                            │
│  4. 需求确认与 PRD                                           │
├─────────────────────────────────────────────────────────────┤
│  阶段二：设计                                                │
│  5. 技术方案设计                                              │
│  6. UI/交互设计确认                                          │
│  7. 设计评审                                                 │
├─────────────────────────────────────────────────────────────┤
│  阶段三：开发                                                │
│  8. 修改前给出计划                                            │
│  9. 编码实现                                                 │
├─────────────────────────────────────────────────────────────┤
│  阶段四：测试                                                │
│  10. 测试计划                                                │
│  11. 测试用例设计                                            │
│  12. 测试执行与验证                                           │
│  13. 测试报告                                                │
├─────────────────────────────────────────────────────────────┤
│  阶段五：发布                                                │
│  14. 代码审查                                                │
│  15. 发布检查与部署                                           │
├─────────────────────────────────────────────────────────────┤
│  阶段六：收尾                                                │
│  16. 文档更新与复盘                                           │
└─────────────────────────────────────────────────────────────┘
```

### 简化版（适用于日常开发）

对于简单任务，可以跳过部分环节：

| 任务类型 | 必须环节 | 可跳过 |
|----------|----------|--------|
| 简单 Bug 修复 | 1-3 → 8-9 → 12 → 14 → 16 | 4-7, 10-11, 13, 15 |
| 样式调整 | 1-3 → 6 → 8-9 → 12 → 16 | 4-5, 7, 10-11, 13, 15 |
| 新功能开发 | 完整流程 | 无 |
| 重构优化 | 1-3 → 5 → 7 → 8-9 → 12 → 14 → 16 | 4, 6, 10-11, 13, 15 |

## 阶段一：需求分析

### 步骤 1：识别需求意图

收到需求后，先判断它属于以下哪一类：

#### A. 新功能开发

用户想新增页面、接口、组件、业务流程、后台能力、数据字段、权限逻辑等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`（如果不知道模块位置）
3. `.ai/CONVENTIONS.md`
4. 和该功能相关的模块文档，例如 `.ai/MODULES/<module>.md`
5. 相关代码入口文件

**不要读取无关模块。**

---

#### B. Bug 修复

用户描述报错、异常行为、线上问题、测试失败、页面不对、接口失败等。

**优先读取：**
1. `CLAUDE.md`
2. 报错堆栈、日志、失败测试中出现的文件
3. `.ai/PROJECT_MAP.md`（如果不知道模块位置）
4. 相关模块文档
5. 相关代码调用链

**先定位问题，再扩大搜索范围。不要直接全项目扫描。**

---

#### C. 重构 / 优化

用户要求优化结构、减少重复、提升性能、调整代码组织、清理技术债等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/ARCHITECTURE.md`
3. `.ai/CONVENTIONS.md`
4. 相关模块文档
5. 被重构区域的代码

**重构前必须说明影响范围，不要顺手修改无关代码。**

---

#### D. UI / 样式修改

用户要求改页面样式、布局、组件状态、交互细节、视觉表现等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/DESIGN_SYSTEM.md`（设计规范：组件库、样式、布局、交互、命名、无障碍）
3. `.ai/PROJECT_MAP.md`（如果不知道页面位置）
4. `.ai/CONVENTIONS.md`
5. 相关页面或组件文件
6. 设计系统、通用组件、样式配置文件

**不要读取后端、数据库、鉴权等无关模块，除非需求明确涉及。**

---

#### E. API / 数据流修改

用户要求新增接口、修改接口字段、调整请求逻辑、处理数据转换、状态管理等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`（如果不知道接口位置）
3. `.ai/ARCHITECTURE.md`
4. 相关模块文档
5. API 封装、类型定义、调用方代码

**必须检查调用方影响，不要只改接口定义。**

---

#### F. 数据库 / Schema / Migration 修改

用户要求新增字段、修改表结构、调整 ORM model、migration、seed 数据等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/ARCHITECTURE.md`
3. 相关模块文档
4. schema / migration / model 相关文件
5. 依赖该字段的业务代码

**必须考虑向后兼容、迁移风险和测试。**

---

#### G. 测试相关

用户要求新增测试、修复测试、提高覆盖率、解释测试失败等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/COMMANDS.md`
3. `.ai/CONVENTIONS.md`
4. 相关测试文件
5. 被测试的源代码文件

**不要读取无关测试目录。**

---

#### H. 构建 / 部署 / 工程配置

用户要求修改 build、CI、lint、format、依赖、环境变量、Docker、部署配置等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/COMMANDS.md`
3. `.ai/ARCHITECTURE.md`
4. package / config / CI / Docker / env 相关文件

**不要修改业务代码，除非配置变更要求联动。**

---

#### I. 文档 / 注释 / README

用户要求写文档、更新说明、补充注释、整理开发指南等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`
3. 目标文档
4. 和文档内容直接相关的少量代码

**不要为了写文档全量读取代码。**

---

#### J. 不确定意图

如果无法判断任务属于哪一类，先读取：

1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`

然后根据项目结构和用户需求判断下一步。

**如果仍然不确定，先向用户提出一个简短澄清问题，不要盲目扫描整个项目。**

### 步骤 3：按需读取上下文

**描述需求后、写代码前，必须读取 `.ai/DESIGN_SYSTEM.md`（设计规范）。**

#### 上下文 token 节省规则

日常开发时，目标是用最少上下文完成任务。

除非任务明显复杂，否则开始修改前的初始读取范围应控制在：

- 1 个默认入口文档：`CLAUDE.md`
- 0 到 2 个 `.ai` 文档
- 1 到 5 个相关代码文件

如果需要超过这个范围，必须先说明原因。

#### 代码搜索规则

优先使用精确搜索，而不是全量读取：

- 搜索函数名
- 搜索组件名
- 搜索路由名
- 搜索接口路径
- 搜索类型名
- 搜索错误信息
- 搜索用户提到的关键词

**优先使用：**

```bash
rg "<keyword>"
find . -name "<filename>"
```

**不要一开始读取整个 `src`、`app`、`pages`、`components`、`lib` 目录。**

#### 按需读取决策树

1. **用户已经提供文件路径、模块名、页面名、组件名、函数名、API 路径或报错信息？**
   - 是 → 优先直接定位相关代码，不必先读 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

2. **已经能判断相关模块？**
   - 是 → 优先读取对应 `.ai/MODULES/<module>.md`，不必先读 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

3. **无法判断模块位置？**
   - 是 → 读取 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

4. **涉及架构、数据流、跨模块调用？**
   - 是 → 读取 `.ai/ARCHITECTURE.md`
   - 否 → 继续下一步

5. **涉及代码风格、命名、测试习惯？**
   - 是 → 读取 `.ai/CONVENTIONS.md`
   - 否 → 继续下一步

6. **涉及开发、测试、构建、部署命令？**
   - 是 → 读取 `.ai/COMMANDS.md`
   - 否 → 继续下一步

7. **最后只读取相关代码文件**

#### 扩大读取范围的条件

只有在以下情况才扩大读取范围：

- 当前信息不足以安全修改
- 搜索结果出现多个同名实现
- 相关逻辑跨多个模块
- 测试失败，需要继续追踪原因
- 文档与代码冲突
- 用户明确要求全面分析

**扩大范围前，必须输出：**

```text
当前缺少的信息：
- <缺少什么>

准备继续读取：
- <文件或目录>

原因：
- <为什么这些内容和当前任务直接相关>
```

#### 禁止行为

- 每次任务默认读取全部 `.ai` 文档
- 每次任务默认读取 `.ai/AI_WORKFLOW.md`
- 每次任务默认读取 `.ai/PROJECT_MAP.md`
- 每次任务默认扫描 `src`
- 每次任务默认扫描 `app`
- 每次任务默认扫描 `components`
- 每次任务默认扫描 `lib`
- 已经定位到相关文件后继续读取无关模块
- 为了"更保险"而读取与当前任务无关的文档
- 描述需求后、写代码前跳过 `.ai/DESIGN_SYSTEM.md` 读取

### 步骤 4：需求确认与 PRD

**触发条件：**
- 新功能开发
- 复杂 Bug 修复（需要明确预期行为）
- 跨模块功能

**跳过条件：**
- 简单样式调整
- 配置变更
- 文档更新

#### 需求分析内容

```text
需求分析：

需求来源：
- <用户反馈 / 业务方需求 / 技术优化 / Bug 修复>

需求目标：
- <解决什么问题>
- <带来什么价值>

用户故事：
- 作为 <角色>，我想要 <功能>，以便 <价值>

验收标准：
- [ ] <标准 1>
- [ ] <标准 2>
- [ ] <标准 3>

影响范围：
- 涉及模块：<列表>
- 涉及用户角色：<列表>
- 数据变更：<是/否>

优先级：
- P0 / P1 / P2

预计工期：
- <时间估算>
```

#### PRD 输出格式（中大型功能）

```text
产品需求文档（PRD）：

功能名称：<功能名>
版本：v1.0
负责人：<开发人员>
日期：<日期>

1. 背景与目标
   <为什么要做这个功能>

2. 用户故事
   - 作为 <角色>，我想要 <功能>，以便 <价值>

3. 功能描述
   <详细描述功能逻辑>

4. 验收标准
   - [ ] <标准 1>
   - [ ] <标准 2>

5. 影响范围
   - 涉及模块：<列表>
   - 数据变更：<说明>

6. 排期
   - 设计：<时间>
   - 开发：<时间>
   - 测试：<时间>
   - 上线：<时间>

用户确认：是 / 否
```

#### 禁止行为

- 跳过需求确认直接开发
- 需求理解不清晰就开始编码
- 忽略验收标准

## 阶段二：设计

### 步骤 5：技术方案设计

**触发条件：**
- 新增完整业务模块
- 跨模块功能开发
- 数据库结构变更
- 架构调整
- 涉及第三方系统集成

**跳过条件：**
- 简单 Bug 修复
- 样式调整
- 单文件修改
- 配置变更

#### 设计内容

##### 1. 影响范围分析

```text
影响范围分析：

涉及模块：
- <模块 1>：<影响点>
- <模块 2>：<影响点>

数据流变化：
- <原流程> → <新流程>

依赖关系：
- <依赖的外部服务/接口>
- <被其他模块依赖的接口>
```

##### 2. 技术方案

```text
技术方案：

实现思路：
- <核心逻辑描述>

关键设计决策：
1. <决策 1>：选择 <方案 A>，原因 <理由>
2. <决策 2>：选择 <方案 B>，原因 <理由>

备选方案：
- 方案 B：<描述>，放弃原因 <理由>
```

##### 3. 数据库设计（如涉及）

```text
数据库变更：

新增表：
- <表名>：<用途>

新增字段：
- <表名>.<字段名>：<类型>：<说明>

索引变更：
- <索引名>：<类型>：<字段>

迁移风险：
- <风险点>：<缓解措施>
```

##### 4. API 设计（如涉及）

```text
API 设计：

新增接口：
- POST /api/v1/<resource>：<用途>
- GET /api/v1/<resource>/:id：<用途>

请求参数：
- <参数名>：<类型>：<必填>：<说明>

响应格式：
{
  "code": 200,
  "data": { ... },
  "message": "success"
}
```

##### 5. 风险评估

```text
风险评估：

技术风险：
- <风险 1>：影响 <程度>，缓解 <措施>

业务风险：
- <风险 1>：影响 <程度>，缓解 <措施>

回滚方案：
- <步骤 1>
- <步骤 2>
```

#### 输出格式

```text
技术方案设计报告：

功能名称：<功能名>
复杂度：简单 / 中等 / 复杂
预计耗时：<时间>

影响范围：
- 涉及模块：<数量>
- 数据库变更：是 / 否
- API 变更：是 / 否
- 跨模块调用：是 / 否

核心设计：
<简要描述实现思路>

关键决策：
1. <决策 1>
2. <决策 2>

风险点：
- <风险 1>
- <风险 2>

用户确认：是 / 否（复杂功能需确认）
```

#### 禁止行为

- 跳过影响范围分析直接写代码
- 复杂功能不输出技术方案
- 忽略数据库变更的迁移风险
- 忽略回滚方案设计

### 步骤 6：UI/交互设计确认

**触发条件：** 新建页面、修改页面布局、新增 UI 组件、修改交互流程时。

**跳过条件：** 仅修改样式细节（颜色、间距）、修复 bug、后端逻辑变更。

#### 流程

1. **输出页面描述**（写代码前）
   - 页面标题、副标题
   - 顶部筛选栏/操作栏
   - 主体内容（表格列/卡片布局/表单）
   - 操作按钮、状态标签
   - 分页、空状态

2. **用户确认**
   - 用户确认描述无误后，再写代码
   - 如有修改，重新确认

3. **设计规范检查**
   - 必须读取 `.ai/DESIGN_SYSTEM.md`
   - 使用 antd 组件库
   - 遵循颜色、字体、间距规范

#### 输出格式

```text
页面描述：
标题：<页面标题>
副标题：<副标题描述>

顶部栏：
- <筛选项 1>（必选/可选）
- <筛选项 2>

主体：
类型：表格 / 卡片 / 表单
列/字段：
- <字段 1>：<说明>
- <字段 2>：<说明>
- 状态标签：<状态映射规则>

操作：
- <按钮 1>：<触发动作>
- <按钮 2>：<触发动作>

底部：分页器

设计规范检查：
- [ ] 使用 antd 组件
- [ ] 遵循颜色规范
- [ ] 无障碍支持
```

- 新增完整业务模块
- 跨模块功能开发
- 数据库结构变更
- 架构调整

**跳过条件：**
- 简单 Bug 修复
- 样式调整
- 单文件修改

```text
设计评审：

技术方案评审：
- [ ] 影响范围分析完整

评审结论：

修改意见：
- <如有>
```

#### 禁止行为

- 跳过设计评审直接开发

## 阶段三：开发

### 步骤 8：修改前给出计划

修改代码前，必须先输出：

```text
计划修改：
1. <文件 1>：<修改原因>
2. <文件 2>：<修改原因>

可能影响：
- <影响点>

验证方式：
- <测试、lint、构建或人工检查方式>
```

#### 禁止行为

- 顺手重构无关代码
- 修改无关格式
- 引入无关依赖
- 修改未涉及模块
- 删除现有逻辑但不说明原因
- 为了通过测试而删除测试或降低校验

#### 16.1 文档更新判断

##### 判断规则

| 变更类型 | 更新文档 | 原因 |
|---|---|---|
| 新增目录、新模块、新入口文件 | `.ai/PROJECT_MAP.md` | 项目结构变化 |
| 新增或修改业务模块规则 | `.ai/MODULES/<module>.md` | 模块知识变化 |
| 改了架构、数据流、跨模块调用方式 | `.ai/ARCHITECTURE.md` | 架构设计变化 |

##### 更新原则

`CLAUDE.md` 只负责 AI 协作核心规则和文档导航，不负责业务细节。

**只有在以下情况才更新 `CLAUDE.md`：**
- 修改了 AI 协作规则
- 修改了读取策略
- 修改了文档分层规则
- 新增了 `.ai/` 下的文档类型

**不要因为以下原因更新 `CLAUDE.md`：**
- 新增了业务模块
- 修改了业务逻辑

4. **更新前先说明计划**

不要直接更新文档，先输出更新计划，让用户确认。

## 最终目标

本项目的 AI 协作目标是：

1. 先识别需求意图
2. 再判断相关模块
3. 只读取必要上下文
4. 中大型功能先做技术方案设计
5. 编码前设计测试用例
6. 最小范围修改代码
7. UI 变更时先确认设计描述
8. 修改后执行测试验证
9. 代码修改后执行审查自查
10. 上线前执行发布检查
11. 完成后判断是否更新 `.ai` 文档

**`CLAUDE.md` 是导航，不是百科全书。**

**日常开发时，`CLAUDE.md` 是唯一默认读取文档。**

**每次代码修改完成后，必须执行测试验证。**

**通过标准：** 所有测试通过，覆盖率 ≥ 70%（新代码）

- 修改了业务逻辑
- 新增了接口
- 修改了数据库模型
- 新增了页面

2. **优先更新模块文档**

如果变更只影响单个模块，优先更新 `.ai/MODULES/<module>.md`，而不是 `.ai/PROJECT_MAP.md` 或 `.ai/ARCHITECTURE.md`。

3. **只更新必要文档**

不要因为"可能有关"就更新文档。只有在变更确实影响了文档内容时才更新。

4. **更新前先说明计划**

不要直接更新文档，先输出更新计划，让用户确认。

## 特殊场景：根据 git diff 更新文档

当用户说："请根据本次 git diff 更新 AI 项目上下文文档"

你必须先查看：

1. `git status --short`
2. `git diff --stat`
3. `git diff --name-status`
4. `git diff --cached --stat`
5. `git diff --cached --name-status`
6. 必要时查看具体 `git diff`
7. 必要时查看具体 `git diff --cached`

**注意：**
- 普通 `git diff` 看不到未跟踪文件
- 普通 `git diff` 不一定包含已暂存改动
- 如果存在未跟踪文件，必须在总结中说明这些文件不会出现在普通 `git diff` 中，并根据文件路径判断是否需要查看内容

然后：

1. 判断本次变更影响了哪些模块、目录、命令、架构或规范
2. 按文档分层规则选择要更新的文档
3. **不要默认更新 `CLAUDE.md`**
4. 更新前先说明计划
5. 只更新必要文档

**固定输出：**

```text
我将根据本次 git diff 更新 AI 项目上下文文档。

本次变更影响：
- <影响点 1>
- <影响点 2>

计划更新：
- <文档 1>：<原因>
- <文档 2>：<原因>

不更新：
- CLAUDE.md：因为本次没有修改 AI 协作规则或项目总入口
```

## 最终目标

本项目的 AI 协作目标是：

1. 先识别需求意图
2. 再判断相关模块
3. 只读取必要上下文
4. 中大型功能先做技术方案设计
5. 编码前设计测试用例
6. 最小范围修改代码
7. UI 变更时先确认设计描述
8. 修改后执行测试验证
9. 代码修改后执行审查自查
10. 上线前执行发布检查
11. 完成后判断是否更新 `.ai` 文档

**`CLAUDE.md` 是导航，不是百科全书。**

**日常开发时，`CLAUDE.md` 是唯一默认读取文档。**

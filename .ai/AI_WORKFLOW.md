# AI 开发工作流

## 重要提醒

**本文档不是日常任务的默认必读文档。**

日常开发时，`CLAUDE.md` 是唯一默认读取文档。

只有在以下情况才读取本文档：
- 初始化 / 重构 AI 文档体系
- 用户要求更新 AI 项目上下文文档
- 用户要求根据 git diff 更新 AI 文档
- 用户要求查看、修改或解释 AI 工作流
- 修改了 AI 协作规则、读取策略或文档分层规则
- 当前流程不明确，仅靠 `CLAUDE.md` 无法判断下一步

---

## 第一步：识别需求意图

收到需求后，先判断它属于以下哪一类：

### A. 新功能开发

用户想新增页面、接口、组件、业务流程、后台能力、数据字段、权限逻辑等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`（如果不知道模块位置）
3. `.ai/CONVENTIONS.md`
4. 和该功能相关的模块文档，例如 `.ai/MODULES/<module>.md`
5. 相关代码入口文件

**不要读取无关模块。**

---

### B. Bug 修复

用户描述报错、异常行为、线上问题、测试失败、页面不对、接口失败等。

**优先读取：**
1. `CLAUDE.md`
2. 报错堆栈、日志、失败测试中出现的文件
3. `.ai/PROJECT_MAP.md`（如果不知道模块位置）
4. 相关模块文档
5. 相关代码调用链

**先定位问题，再扩大搜索范围。不要直接全项目扫描。**

---

### C. 重构 / 优化

用户要求优化结构、减少重复、提升性能、调整代码组织、清理技术债等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/ARCHITECTURE.md`
3. `.ai/CONVENTIONS.md`
4. 相关模块文档
5. 被重构区域的代码

**重构前必须说明影响范围，不要顺手修改无关代码。**

---

### D. UI / 样式修改

用户要求改页面样式、布局、组件状态、交互细节、视觉表现等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`（如果不知道页面位置）
3. `.ai/CONVENTIONS.md`
4. 相关页面或组件文件
5. 设计系统、通用组件、样式配置文件

**不要读取后端、数据库、鉴权等无关模块，除非需求明确涉及。**

---

### E. API / 数据流修改

用户要求新增接口、修改接口字段、调整请求逻辑、处理数据转换、状态管理等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`（如果不知道接口位置）
3. `.ai/ARCHITECTURE.md`
4. 相关模块文档
5. API 封装、类型定义、调用方代码

**必须检查调用方影响，不要只改接口定义。**

---

### F. 数据库 / Schema / Migration 修改

用户要求新增字段、修改表结构、调整 ORM model、migration、seed 数据等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/ARCHITECTURE.md`
3. 相关模块文档
4. schema / migration / model 相关文件
5. 依赖该字段的业务代码

**必须考虑向后兼容、迁移风险和测试。**

---

### G. 测试相关

用户要求新增测试、修复测试、提高覆盖率、解释测试失败等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/COMMANDS.md`
3. `.ai/CONVENTIONS.md`
4. 相关测试文件
5. 被测试的源代码文件

**不要读取无关测试目录。**

---

### H. 构建 / 部署 / 工程配置

用户要求修改 build、CI、lint、format、依赖、环境变量、Docker、部署配置等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/COMMANDS.md`
3. `.ai/ARCHITECTURE.md`
4. package / config / CI / Docker / env 相关文件

**不要修改业务代码，除非配置变更要求联动。**

---

### I. 文档 / 注释 / README

用户要求写文档、更新说明、补充注释、整理开发指南等。

**优先读取：**
1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`
3. 目标文档
4. 和文档内容直接相关的少量代码

**不要为了写文档全量读取代码。**

---

### J. 不确定意图

如果无法判断任务属于哪一类，先读取：

1. `CLAUDE.md`
2. `.ai/PROJECT_MAP.md`

然后根据项目结构和用户需求判断下一步。

**如果仍然不确定，先向用户提出一个简短澄清问题，不要盲目扫描整个项目。**

## 第三步：按需读取上下文

### 上下文 token 节省规则

日常开发时，目标是用最少上下文完成任务。

除非任务明显复杂，否则开始修改前的初始读取范围应控制在：

- 1 个默认入口文档：`CLAUDE.md`
- 0 到 2 个 `.ai` 文档
- 1 到 5 个相关代码文件

如果需要超过这个范围，必须先说明原因。

### 代码搜索规则

优先使用精确搜索，而不是全量读取：

- 搜索函数名
- 搜索组件名
- 搜索路由名
- 搜索接口路径
- 搜索类型名
- 搜索错误信息
- 搜索用户提到的关键词

**优先使用：**

```bash
rg "<keyword>"
find . -name "<filename>"
```

**不要一开始读取整个 `src`、`app`、`pages`、`components`、`lib` 目录。**

### 按需读取决策树

1. **用户已经提供文件路径、模块名、页面名、组件名、函数名、API 路径或报错信息？**
   - 是 → 优先直接定位相关代码，不必先读 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

2. **已经能判断相关模块？**
   - 是 → 优先读取对应 `.ai/MODULES/<module>.md`，不必先读 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

3. **无法判断模块位置？**
   - 是 → 读取 `.ai/PROJECT_MAP.md`
   - 否 → 继续下一步

4. **涉及架构、数据流、跨模块调用？**
   - 是 → 读取 `.ai/ARCHITECTURE.md`
   - 否 → 继续下一步

5. **涉及代码风格、命名、测试习惯？**
   - 是 → 读取 `.ai/CONVENTIONS.md`
   - 否 → 继续下一步

6. **涉及开发、测试、构建、部署命令？**
   - 是 → 读取 `.ai/COMMANDS.md`
   - 否 → 继续下一步

7. **最后只读取相关代码文件**

### 扩大读取范围的条件

只有在以下情况才扩大读取范围：

- 当前信息不足以安全修改
- 搜索结果出现多个同名实现
- 相关逻辑跨多个模块
- 测试失败，需要继续追踪原因
- 文档与代码冲突
- 用户明确要求全面分析

**扩大范围前，必须输出：**

```text
当前缺少的信息：
- <缺少什么>

准备继续读取：
- <文件或目录>

原因：
- <为什么这些内容和当前任务直接相关>
```

### 禁止行为

- 每次任务默认读取全部 `.ai` 文档
- 每次任务默认读取 `.ai/AI_WORKFLOW.md`
- 每次任务默认读取 `.ai/PROJECT_MAP.md`
- 每次任务默认扫描 `src`
- 每次任务默认扫描 `app`
- 每次任务默认扫描 `components`
- 每次任务默认扫描 `lib`
- 已经定位到相关文件后继续读取无关模块
- 为了"更保险"而读取与当前任务无关的文档

## 第四步：修改前给出计划

修改代码前，必须先输出：

```text
计划修改：
1. <文件 1>：<修改原因>
2. <文件 2>：<修改原因>

可能影响：
- <影响点>

验证方式：
- <测试、lint、构建或人工检查方式>
```

### 禁止行为

- 顺手重构无关代码
- 修改无关格式
- 引入无关依赖
- 修改未涉及模块
- 删除现有逻辑但不说明原因
- 为了通过测试而删除测试或降低校验
- 在未说明风险的情况下修改公共 API 或数据结构

## 第五步：完成后判断是否更新 AI 文档

每次完成需求后，都要判断是否需要更新 AI 项目上下文文档。

**不要默认更新 `CLAUDE.md`。**

### 判断规则

| 变更类型 | 更新文档 | 原因 |
|---|---|---|
| 新增目录、新模块、新入口文件 | `.ai/PROJECT_MAP.md` | 项目结构变化 |
| 新增或修改业务模块规则 | `.ai/MODULES/<module>.md` | 模块知识变化 |
| 改了架构、数据流、跨模块调用方式 | `.ai/ARCHITECTURE.md` | 架构设计变化 |
| 改了代码风格、命名、测试习惯 | `.ai/CONVENTIONS.md` | 编码规范变化 |
| 新增或修改开发、测试、构建命令 | `.ai/COMMANDS.md` | 命令变化 |
| 改了 AI 协作规则、读取策略、文档分层规则 | `CLAUDE.md` | 协作规则变化 |
| 普通功能代码变更，没有新增长期知识 | 不需要更新 | 无需持久化 |

### 输出格式

完成代码修改后，输出：

```text
AI 文档更新判断：

是否需要更新文档：是 / 否

原因：
<说明原因>

如果需要更新，计划更新：
- <文档 1>：<原因>
- <文档 2>：<原因>

不会更新：
- CLAUDE.md：<原因>
```

### 更新原则

1. **不要默认更新 `CLAUDE.md`**

`CLAUDE.md` 只负责 AI 协作核心规则和文档导航，不负责业务细节。

**只有在以下情况才更新 `CLAUDE.md`：**
- 修改了 AI 协作规则
- 修改了读取策略
- 修改了文档分层规则
- 新增了 `.ai/` 下的文档类型

**不要因为以下原因更新 `CLAUDE.md`：**
- 新增了业务模块
- 修改了业务逻辑
- 新增了接口
- 修改了数据库模型
- 新增了页面

2. **优先更新模块文档**

如果变更只影响单个模块，优先更新 `.ai/MODULES/<module>.md`，而不是 `.ai/PROJECT_MAP.md` 或 `.ai/ARCHITECTURE.md`。

3. **只更新必要文档**

不要因为"可能有关"就更新文档。只有在变更确实影响了文档内容时才更新。

4. **更新前先说明计划**

不要直接更新文档，先输出更新计划，让用户确认。

## 特殊场景：根据 git diff 更新文档

当用户说："请根据本次 git diff 更新 AI 项目上下文文档"

你必须先查看：

1. `git status --short`
2. `git diff --stat`
3. `git diff --name-status`
4. `git diff --cached --stat`
5. `git diff --cached --name-status`
6. 必要时查看具体 `git diff`
7. 必要时查看具体 `git diff --cached`

**注意：**
- 普通 `git diff` 看不到未跟踪文件
- 普通 `git diff` 不一定包含已暂存改动
- 如果存在未跟踪文件，必须在总结中说明这些文件不会出现在普通 `git diff` 中，并根据文件路径判断是否需要查看内容

然后：

1. 判断本次变更影响了哪些模块、目录、命令、架构或规范
2. 按文档分层规则选择要更新的文档
3. **不要默认更新 `CLAUDE.md`**
4. 更新前先说明计划
5. 只更新必要文档

**固定输出：**

```text
我将根据本次 git diff 更新 AI 项目上下文文档。

本次变更影响：
- <影响点 1>
- <影响点 2>

计划更新：
- <文档 1>：<原因>
- <文档 2>：<原因>

不更新：
- CLAUDE.md：因为本次没有修改 AI 协作规则或项目总入口
```

## 文档更新示例

### 示例 1：新增业务模块

**变更：**
- 新增 `internal/service/notification_service.go`
- 新增 `internal/api/notification_handlers.go`
- 新增 `frontend/src/pages/Notification.tsx`

**判断：**

```text
AI 文档更新判断：

是否需要更新文档：是

原因：
新增了通知模块，需要更新项目结构索引和新增模块文档。

计划更新：
- .ai/PROJECT_MAP.md：新增通知模块的目录结构和路由
- .ai/MODULES/notification.md：新增通知模块的业务规则说明

不会更新：
- CLAUDE.md：因为本次没有修改 AI 协作规则，只是新增业务模块
```

### 示例 2：修改架构

**变更：**
- 在 Service 层和 Repository 层之间新增 Cache 层
- 修改了所有 Service 的调用方式

**判断：**

```text
AI 文档更新判断：

是否需要更新文档：是

原因：
新增了 Cache 层，改变了分层架构和数据流。

计划更新：
- .ai/ARCHITECTURE.md：更新分层架构图，说明 Cache 层的职责和调用方式

不会更新：
- CLAUDE.md：因为本次没有修改 AI 协作规则，只是修改了架构设计
- .ai/PROJECT_MAP.md：因为没有新增目录，只是修改了调用方式
```

### 示例 3：普通功能开发

**变更：**
- 修改 `internal/service/user_service.go` 的 `GetUser` 方法
- 修改 `frontend/src/pages/UserList.tsx` 的列表展示

**判断：**

```text
AI 文档更新判断：

是否需要更新文档：否

原因：
本次只是修改了现有功能的实现细节，没有新增模块、没有修改架构、没有修改规范、没有新增命令，不需要更新 AI 文档。

不会更新：
- CLAUDE.md：因为本次没有修改 AI 协作规则
- .ai/PROJECT_MAP.md：因为没有新增目录或模块
- .ai/ARCHITECTURE.md：因为没有修改架构或数据流
- .ai/CONVENTIONS.md：因为没有修改编码规范
- .ai/COMMANDS.md：因为没有修改命令
- .ai/MODULES/org.md：因为只是修改了实现细节，没有修改业务规则
```

## 最终目标

本项目的 AI 协作目标是：

1. 先识别需求意图
2. 再判断相关模块
3. 只读取必要上下文
4. 最小范围修改代码
5. 修改后验证或说明无法验证的原因
6. 完成后判断是否更新 `.ai` 文档

**`CLAUDE.md` 是导航，不是百科全书。**

**日常开发时，`CLAUDE.md` 是唯一默认读取文档。**
